# AI 기반 MSDS 관리대장 자동화 및 변경감시 앱 기획안

## 1. 한 줄 요약

> **사업장에서 보유한 MSDS를 로컬 환경에서 자동으로 정리하고, 제품별·현장별 사용 현황, 개정 필요 여부, CAS No. 기반 검토 필요 항목, 엑셀 관리대장, 공급사 요청 리스트를 만들어주는 보건관리자용 업무보조 앱.**

이 앱은 AI가 유해화학물질 여부를 최종 판정하는 시스템이 아닙니다.  
보건관리자와 간호사가 MSDS를 더 빠르고 정확하게 관리할 수 있도록 돕는 **검수·정리·알림·엑셀 자동화 도구**입니다.

---

## 2. 제안 배경

현재 사업장에서는 여러 현장에서 다양한 화학제품과 자재를 사용하고 있습니다. 이에 따라 관리해야 할 MSDS, 즉 물질안전보건자료의 종류도 많습니다.

특히 하나의 제품이 한 현장에서만 사용되는 것이 아니라, **여러 현장에서 동시에 사용되는 경우**가 많습니다. 따라서 특정 제품의 MSDS가 개정되거나, 구성성분 중 하나가 새롭게 규제 대상 또는 검토 대상이 될 경우, 해당 제품을 사용하는 모든 현장에 영향을 줄 수 있습니다.

현재 MSDS 관리는 다음과 같은 어려움이 있습니다.

| 구분 | 현재 어려움 |
|---|---|
| MSDS 수량 | 현장 수가 많고 제품 종류가 많아 관리 대상이 많음 |
| 개정관리 | 공급사 MSDS 개정 여부를 일일이 확인해야 함 |
| 성분 확인 | CAS No., 성분명, 함유량, 영업비밀 여부 확인이 번거로움 |
| 현장 매핑 | 어떤 제품이 어느 현장에서 사용되는지 한눈에 보기 어려움 |
| 법규 검토 | 유해화학물질, 산안법, 화관법 관련 검토가 수작업에 의존 |
| 엑셀 관리 | 관리대장, 현장별 목록, 공급사 요청 리스트 작성이 반복 업무 |
| 시스템 등록 | 사내 시스템 등록 전 필요한 정보를 다시 정리해야 함 |
| 점검 대응 | 최신본 여부, 게시 여부, 제출번호, 개정일 등을 빠르게 확인하기 어려움 |

따라서 이 앱은 단순한 MSDS 보관함이 아니라, **현장별 MSDS 관리대장과 검토 필요 항목을 자동으로 정리하는 업무보조 도구**로 기획합니다.

---

## 3. 앱의 목적

이 앱의 목적은 **AI가 법적 판단을 대신하는 것**이 아닙니다.

목적은 다음과 같습니다.

> **보건관리자가 MSDS를 더 빠르고 정확하게 관리할 수 있도록, MSDS 정보 추출, 현장별 관리, 변경감시, 엑셀 정리, 공급사 요청 업무를 자동화하는 것.**

즉, 최종 판단은 보건관리자가 하되, 앱은 다음 업무를 도와줍니다.

1. MSDS PDF에서 제품명, 공급사, 개정일, 제출번호, CAS No., 성분명, 함유량 등을 자동 추출
2. 제품별 사용 현장을 연결하여 영향 범위 확인
3. CAS No. 기준으로 공식 DB와 비교하여 검토 필요 여부 표시
4. 최신 MSDS 요청 대상 자동 정리
5. 현장별 MSDS 관리대장 엑셀 자동 생성
6. 공급사 요청 메일 초안 자동 생성
7. 개정 필요, 제출번호 누락, CAS No. 누락, 영업비밀 성분 등 검토 필요 항목 알림

---

## 4. 왜 필요한가?

MSDS 관리는 단순한 문서 보관 업무가 아닙니다.

MSDS에는 화학제품의 구성성분, 함유량, 유해성·위험성, 취급 및 저장 방법, 보호구, 법적 규제현황 등이 포함됩니다. 따라서 MSDS가 오래되었거나, 구성성분이 제대로 확인되지 않거나, 현장 게시본과 실제 사용 제품이 맞지 않으면 점검 대응과 보건관리 업무에 문제가 생길 수 있습니다.

특히 앞으로는 다음 항목을 체계적으로 관리할 필요가 있습니다.

| 확인 항목 | 필요한 이유 |
|---|---|
| 최신 개정본 여부 | 오래된 MSDS 사용 방지 |
| MSDS 제출번호 | 향후 점검 대응에서 중요해질 수 있는 항목 |
| 구성성분 CAS No. | 법규 DB와 매칭하기 위한 핵심 키 |
| 함유량/농도 범위 | 규제 대상 여부 검토에 필요 |
| 영업비밀/대체명칭 | 자동판정 불가, 공급사 확인 필요 |
| 사용 현장 | 변경 발생 시 영향 범위 확인 |
| 공급사 정보 | 최신본 요청 및 확인 필요 |
| 현장 게시 여부 | 현장별 MSDS 관리상태 확인 필요 |

## 5. 이전 실패 앱과 다르게 접근해야 하는 이유

이전에 검토했던 “안전 법령 진단 도우미 앱”은 대한민국 전체 법령을 대상으로 자연어 질문을 입력하면 관련 법령을 찾아 답변하는 구조였습니다.

하지만 이 방식은 다음과 같은 문제가 있었습니다.

- 검색 범위가 너무 넓음
- 사용자의 질문이 모호하면 관련 법령을 정확히 찾기 어려움
- AI가 엉뚱한 법령을 찾아올 위험이 큼
- 많은 수의 법령을 검색하면서 응답 시간이 길어짐
- MVP 단계에서 너무 많은 기능을 넣어 앱이 무거워짐

이번 MSDS 앱은 그 방식으로 가면 안 됩니다.

즉, 이 앱은 **질문형 법규 챗봇**이 아니라, **정해진 MSDS 문서에서 정해진 값을 추출하고, CAS No.와 농도 기준으로 공식 DB와 비교하는 관리 도구**로 설계해야 합니다.

| 구분 | 실패 위험이 큰 방식 | 이번 앱의 권장 방식 |
|---|---|---|
| 핵심 방식 | AI에게 법규 질문 | MSDS 데이터 구조화 |
| 검색 범위 | 대한민국 전체 법령 | 우리 사업장 사용 제품 |
| 판단 방식 | AI 자연어 추론 | CAS No. + 공식 DB + 규칙 비교 |
| 결과물 | 법령 답변 | 관리대장, 검토 필요 알림, 엑셀 |
| 최종 판단 | AI가 답변 | 보건관리자가 검수 |
| 실패 위험 | 엉뚱한 법령 검색 | 추출값 오류 중심, 관리 가능 |

따라서 이번 앱의 핵심 원칙은 다음과 같습니다.

> **AI는 판단자가 아니라 보조자다.**  
> **법규 최종판정이 아니라 검토 필요 항목을 찾아주는 도구다.**  
> **챗봇보다 표, 대장, 알림, 엑셀 출력이 핵심이다.**

## 7. 핵심 기능

## 7.1 MSDS PDF 자동 분석

사용자가 MSDS PDF를 업로드하면 앱이 핵심 정보를 자동으로 추출합니다.

| 항목 | 예시 |
|---|---|
| 제품명 | A세척제 |
| 공급사/제조사 | ○○케미칼 |
| MSDS 개정일 | 2025-12-01 |
| MSDS 버전 | Rev. 5 |
| MSDS 제출번호 | AA00000-0000000000 |
| 구성성분명 | Acetone, Toluene 등 |
| CAS No. | 67-64-1 등 |
| 함유량/농도 | 10~20%, <1%, 영업비밀 등 |
| GHS 분류 | 인화성 액체, 피부 자극성 등 |
| 신호어 | 위험/경고 |
| 그림문자 | 불꽃, 느낌표 등 |
| 법적 규제현황 | 산안법, 화관법, 위험물 등 |
| 보호구 정보 | 보안경, 내화학장갑, 방독마스크 등 |

초기 버전에서는 모든 항목을 완벽히 처리하기보다, 아래 항목을 우선 추출합니다.

```text
제품명
공급사
MSDS 개정일
MSDS 버전
MSDS 제출번호
CAS No.
성분명
함유량/농도
영업비밀 여부
사용현장
```

## 7.2 추출값 검수 화면

AI 또는 파서가 추출한 정보는 사람이 확인해야 합니다.

따라서 앱은 추출값만 보여주는 것이 아니라, **원문 근거와 함께 보여주는 검수 화면**이 필요합니다.

예시:

```text
추출값: CAS 67-64-1
원문 위치: 3. 구성성분의 명칭 및 함유량
원문 내용: Acetone 30~60% CAS 67-64-1
검수 상태: 확인 / 수정 / 제외
```

이 화면이 있어야 보건관리자가 AI 추출 결과를 신뢰할 수 있습니다.

## 7.3 제품별 사용 현장 관리

하나의 제품은 여러 현장에서 사용할 수 있으므로, 앱은 제품과 현장을 1:1로 연결하지 않고 **N:M 구조**로 관리합니다.

예를 들어 A제품이 여러 현장에서 사용된다면 다음과 같이 관리합니다.

| 제품명 | 사용 현장 | 사용 목적 | 보관 장소 | 담당부서 |
|---|---|---|---|---|
| A세척제 | 1공장 | 설비 세척 | 약품보관장 | 생산1팀 |
| A세척제 | 3공장 | 부품 세척 | 정비실 | 정비팀 |
| A세척제 | 분석실 | 시험 전처리 | 시약장 | 품질팀 |

이렇게 관리하면 A제품에 변경사항이 생겼을 때, 영향을 받는 현장을 바로 확인할 수 있습니다.

예시 알림:

```text
[검토 필요] A세척제

사유:
- MSDS 개정일이 오래됨
- 구성성분 CAS No. 중 일부 공식 DB 변경 후보 감지
- 제출번호 누락

영향 현장:
- 1공장
- 3공장
- 분석실

필요 조치:
1. 공급사 최신 MSDS 요청
2. 현장 게시 MSDS 교체 여부 확인
3. 사내 시스템 등록정보 수정 여부 확인
4. 해당 현장 담당자에게 공유
```

## 7.4 CAS No. 기준 공식 DB 매칭

AI가 성분명을 보고 임의로 판단하는 방식은 위험합니다.

따라서 핵심 기준은 **CAS No.** 입니다.

앱은 MSDS에서 추출한 CAS No.를 기준으로 공식 DB를 조회하고, 현재 우리 대장에 저장된 정보와 비교합니다.

활용 가능한 공식 데이터 예시는 다음과 같습니다.

| 데이터 | 활용 목적 |
|---|---|
| KOSHA 화학물질정보/MSDS 정보 | 산안법 MSDS 관련 참고 정보 확인 |
| 한국환경공단 화학물질 정보 조회 서비스 | CAS번호, KE번호, 물질분류, 함량정보, 예외정보, 고시일자, 고시정보 확인 |
| 공공데이터포털 API | 공식 DB 기반 자동 조회 |

앱은 공식 DB를 기준으로 **최종 판정**을 하는 것이 아니라, 다음과 같이 표시합니다.

```text
판정: 검토 필요
사유: 공식 DB 기준 규제정보 변경 후보가 감지됨
필요 조치: 공급사 최신 MSDS 및 해당 여부 확인 필요
```

## 7.5 변경감시 기능

공식 DB 전체를 주기적으로 저장할 필요는 없습니다.

대신 우리 사업장에서 사용하는 제품의 **CAS No. 목록**, 즉 watchlist만 관리합니다.

예를 들어 우리 사업장에서 사용하는 CAS No.가 150개라면, 앱은 이 150개에 대해서만 주기적으로 공식 DB를 조회하고 이전 조회 결과와 비교합니다.

```text
우리 사업장 관심 CAS 목록
        ↓
공식 DB 재조회
        ↓
이전 조회 결과와 비교
        ↓
변경 후보 감지
        ↓
영향 제품 및 현장 표시
```

변경감시가 필요한 이유는 다음과 같습니다.

| 이유 | 설명 |
|---|---|
| 변경 인지 | 공식 DB가 바뀌어도 별도 알림이 오지 않을 수 있음 |
| 영향 분석 | 변경된 CAS가 어떤 제품과 현장에 영향을 주는지 확인 필요 |
| 이력 관리 | 언제 변경을 인지했고, 언제 조치했는지 남겨야 함 |
| 공급사 확인 | 공식 DB와 공급사 MSDS가 다를 경우 최신본 요청 필요 |

알림 예시는 다음과 같습니다.

```text
[규제정보 변경 후보]

CAS No.: 000-00-0
관련 제품: A세척제, B희석제
영향 현장: 1공장, 4공장, 분석실

변경 내용:
- 공식 DB 고시정보 또는 함량정보 변경 후보 감지

필요 조치:
1. 공급사 최신 MSDS 요청
2. 유해화학물질 해당 여부 확인
3. 사내 시스템 등록정보 수정 필요 여부 확인
4. 현장 게시 MSDS 교체 필요 여부 확인
```

## 7.6 엑셀 자동 출력

실무에서는 결국 엑셀 자료가 필요합니다.

따라서 이 앱은 웹 화면만 제공하는 것이 아니라, 보건관리자가 바로 사용할 수 있는 엑셀 양식을 자동으로 생성해야 합니다.

출력 가능한 엑셀은 다음과 같습니다.

| 출력물 | 용도 |
|---|---|
| 전체 MSDS 관리대장 | 전체 제품 현황 관리 |
| 현장별 MSDS 목록 | 현장 게시본 확인 |
| 개정 필요 MSDS 목록 | 최신본 요청 대상 확인 |
| 제출번호 누락 목록 | 점검 대비 |
| CAS No. 누락/영업비밀 목록 | 공급사 확인 필요 대상 |
| 유해화학물질 검토 필요 목록 | 화관법 등 검토 대상 |
| 공급사 요청 리스트 | 최신 MSDS 요청 메일 발송용 |

예시 출력 컬럼:

| 제품명 | 공급사 | 사용현장 | MSDS 개정일 | 제출번호 | 주요 CAS No. | 농도 | 상태 | 필요조치 |
|---|---|---|---|---|---|---|---|---|
| A세척제 | ○○케미칼 | 1공장, 3공장 | 2024-02-01 | 없음 | 67-64-1 | 10~20% | 검토 필요 | 제출번호 확인, 최신본 요청 |

## 7.7 공급사 요청 메일 자동 생성

MSDS 개정 필요, 제출번호 누락, CAS No. 누락, 영업비밀 성분 확인 등이 필요한 경우, 앱이 공급사 요청 메일 초안을 자동으로 생성합니다.

예시:

```text
제목: [MSDS 최신본 요청] A세척제 MSDS 및 유해화학물질 해당 여부 확인 요청

안녕하세요.

당사에서 사용 중인 귀사 제품 A세척제와 관련하여 MSDS 최신본 확인을 요청드립니다.

확인 요청사항은 아래와 같습니다.

1. 최신 MSDS 개정본
2. MSDS 개정일 및 버전
3. MSDS 제출번호 기재 여부
4. 구성성분 CAS No. 및 함유량 범위
5. 화학물질관리법상 유해화학물질 해당 여부
6. 영업비밀 또는 대체명칭 사용 시 승인 여부

확인 후 회신 부탁드립니다.

감사합니다.
```

이 기능은 보건관리자의 반복적인 메일 작성 업무를 줄이는 데 도움이 됩니다.

## 8. 앱의 운영 방식

이 앱은 사내 보안 이슈를 고려하여 **로컬 기반**으로 운영하는 것을 원칙으로 합니다.

즉, MSDS 원본 파일과 제품 정보는 외부 클라우드로 보내지 않고, 사내 PC 또는 사내 서버에서 관리합니다.

운영 구조는 다음과 같습니다.

```text
사내 PC 또는 Mac mini
        ↓
로컬 웹앱
        ↓
로컬 DB
        ↓
MSDS PDF 저장소
        ↓
엑셀 출력
        ↓
공식 DB 조회 또는 수동 업데이트
```

보안 수준에 따라 두 가지 방식이 가능합니다.

| 방식 | 설명 |
|---|---|
| 완전 로컬 방식 | 외부망 없이 MSDS와 대장을 관리하고, 공식 DB 업데이트 파일만 수동 반입 |
| 제한적 외부조회 방식 | MSDS 원문은 로컬 저장, 외부로는 CAS No.만 공식 API 조회 |

초기에는 보안상 부담이 적은 **완전 로컬 또는 제한적 외부조회 방식**으로 시작하는 것이 적절합니다.

### 8.1 MVP 검증용 AI 처리 구조

1차 MVP에서는 비용을 최소화하고 빠르게 가능성을 검증하기 위해, **Mac mini를 24/7 로컬 서버로 운영하고 로컬 웹앱에서 Codex CLI를 호출하는 방식**을 우선 검토합니다.

이 방식의 목적은 상용 운영 구조를 확정하는 것이 아니라, 다음 가설을 빠르게 확인하는 것입니다.

> **MSDS PDF에서 사내 시스템 등록에 필요한 후보값을 추출하고, 담당자가 원문 근거를 보며 검수하면 실제 등록 업무 시간이 줄어드는가?**

MVP 단계의 처리 흐름은 다음과 같습니다.

```text
MSDS PDF 업로드
        ↓
로컬 PDF 텍스트/표 추출
        ↓
Codex CLI에 추출 텍스트 + 사내 등록 필드 스키마 전달
        ↓
JSON 형태의 등록 후보값 생성
        ↓
원문 근거와 함께 검수 화면 표시
        ↓
담당자 확인/수정
        ↓
사내 시스템 등록용 엑셀 또는 복사/붙여넣기 데이터 생성
```

이때 Codex CLI는 PDF 원문을 법적으로 판정하는 주체가 아니라, **추출된 텍스트를 사내 등록 필드 구조에 맞춰 정리하는 임시 AI 실행 엔진**으로 사용합니다.

법적 규제 항목은 빈칸으로 방치하지 않고, 아래 상태 중 하나로 반드시 표시합니다.

```text
해당
비해당
해당 후보
비해당 후보
확인필요
공급사 확인 필요
내부 기준 확인 필요
```

즉, 공식 출처로 명확히 확인되는 값은 해당/비해당으로 표시하고, 확정이 어려운 값은 “확인필요”로 남기되 **조회한 출처, 조회일, 근거, 다음 조치**를 함께 표시합니다.  
이 원칙은 “AI가 값을 억지로 확정하는 위험”과 “모르는 항목을 빈칸으로 방치하는 문제”를 동시에 줄이기 위한 MVP UX 원칙입니다.

운영 안정성과 제품화를 고려해 AI 호출부는 처음부터 교체 가능한 구조로 둡니다.

| 구성요소 | MVP 방식 | 향후 교체 가능 대상 |
|---|---|---|
| AI 실행 엔진 | Codex CLI | OpenAI API, 로컬 LLM, 사내망 모델 |
| PDF 처리 | 로컬 텍스트/표 추출 | OCR, 문서 AI, 전문 PDF 파서 |
| 규제정보 확인 | MSDS 원문 힌트 + 수동 검수 | CAS No. 기반 공식 DB/API, 내부 룰셋 |
| 최종 확정 | 담당자 검수 | 동일하게 담당자 검수 유지 |

따라서 MVP에서는 Codex CLI를 적극 활용하되, 앱 전체가 특정 CLI에 종속되지 않도록 **AI 어댑터 계층**을 둡니다.  
추후 비용, 속도, 보안, 안정성 요구가 명확해지면 이 계층만 OpenAI API, 로컬 LLM, 사내 전용 모델 등으로 교체할 수 있게 설계합니다.

## 9. 앱이 해주는 것과 해주지 않는 것

이 앱은 법적 책임을 대신지는 시스템이 아닙니다.

명확하게 역할을 구분해야 합니다.

### 앱이 해주는 것

|---|---|

| 현장별 연결 | 제품이 어느 현장에서 쓰이는지 관리 |
| 검토 필요 알림 | 개정일, 제출번호, CAS 누락, 규제변경 후보 표시 |
| 공식 DB 비교 | CAS No. 기준 참고 정보 조회 |
| 엑셀 자동 생성 | 관리대장, 요청 리스트 자동 출력 |
| 공급사 요청 초안 | 최신 MSDS 요청 메일 작성 보조 |
| 이력 관리 | 변경 인지일, 조치일, 담당자 관리 |

### 앱이 해주지 않는 것

| 제외 기능 | 이유 |
|---|---|
| 유해화학물질 최종판정 | 법적 판단은 담당자 검토 필요 |
| MSDS 자동 작성 | 제조·수입자 책임 영역 |
| 영업비밀 성분 추정 | 추정 금지, 공급사 확인 필요 |
| 사내 시스템 자동 등록 | 초기 MVP에서는 오류·보안 리스크 큼 |
| 전체 법령 Q&A 챗봇 | 이전 실패 사례와 유사한 리스크 |
| 모든 스캔본 OCR 완벽 처리 | 초기 난이도 과다. 단, 스캔본 감지와 제한적 로컬 OCR 시도, 실패 시 수동입력/공급사 텍스트 PDF 요청 흐름은 포함 |

앱 화면에는 다음 문구를 표시하는 것이 좋습니다.

```text
본 시스템은 MSDS 관리 및 검토 보조 도구입니다.
최종 법적 판단과 사내 시스템 등록은 담당자의 검토를 거쳐 진행해야 합니다.
공식 DB 정보는 참고용이며, 최종 MSDS는 공급사 제공본을 기준으로 확인해야 합니다.
```

## 10. 1차 MVP 개발 범위

처음부터 모든 기능을 만들면 실패 가능성이 높습니다.

따라서 1차 버전은 다음 범위로 제한합니다.

### 1차 MVP 포함 기능

| 기능 | 설명 |
|---|---|
| MSDS PDF 업로드 | 파일 등록 및 보관 |
| 사내 등록 필드 후보 추출 | 제품명, 공급사, 개정일, 제출번호, CAS No., 성분명, 함유량, 법적 규제 항목 후보 |
| Codex CLI 기반 구조화 | MVP 검증 단계에서 로컬 추출 텍스트를 사내 등록 스키마에 맞는 JSON 후보값으로 정리 |
| 사용자 검수 화면 | AI/파서가 추출한 값을 원문 근거와 함께 사람이 확인·수정 |
| 사내 등록 화면 기준 스키마 | 물품 기본 정보, 함유량 및 법적규제 정보, 유해성 위험성 정보, 개정이력 순서로 검수 화면 구성 |

| 공급사 요청 메일 초안 | 요청 메일 자동 생성 |

### 1차 MVP 제외 기능

| 제외 기능 | 제외 이유 |

   - 성분 확인
   - 현장별 정리
   - 시스템 등록
   - 점검 대응
   - 공급사 회신 관리

8. 스캔본 MSDS가 많은지, 텍스트 PDF가 많은지
9. 공급사에 최신 MSDS 요청하는 표준 메일 양식이 있는지

## 18. 사업화 가능성 평가

이 앱은 내부 업무개선 도구로는 매우 적합합니다.

다른 회사의 보건관리자나 보건간호사에게도 어필 가능성이 있습니다. 특히 다음과 같은 사업장에는 수요가 있을 가능성이 큽니다.

```text
- 현장이 여러 개다
- MSDS가 100개 이상이다
- 공급사가 많다
- 세척제, 윤활유, 시약, 도료, 접착제, 분체, 산·알칼리류가 많다
- 보건관리자가 엑셀로 관리한다
- 최신본 요청과 개정일 확인이 수작업이다
- 점검 때 MSDS 게시/교육/경고표지로 지적받는다
- 사내 시스템과 실제 현장 사용물질이 잘 맞지 않는다
```

다만 외부 월구독 서비스로 확장하려면 단순 PDF → 엑셀 변환기만으로는 부족합니다.

월구독이 되려면 다음 기능이 필요합니다.

| 월구독 핵심 기능 | 설명 |
|---|---|
| 규제정보 변경감시 | 주기적으로 관심 CAS No.를 확인하고 변경 후보 알림 |
| 공급사 최신본 요청 관리 | 요청, 회신, 미회신 관리 |
| 현장별 영향 알림 | 어떤 제품이 어느 현장에 영향을 주는지 표시 |
| 월간 점검 리포트 | 점검 대응용 보고서 자동 생성 |
| 고객사 엑셀 양식 맞춤 | 회사별 양식에 맞게 출력 |
| 로컬/온프레미스 설치 | MSDS 원문을 외부로 보내지 않는 구조 |

따라서 사업화 방향은 일반 클라우드 SaaS보다 **로컬 설치형 또는 온프레미스 구독형 서비스**가 더 현실적입니다.

```text
고객사 내부에 앱 설치
+ 규제 DB 업데이트팩 제공
+ 엑셀 양식 커스터마이징
+ 유지보수
+ 월/연 구독 과금
```

> **“대형 솔루션까지는 부담스럽지만, 엑셀로 MSDS를 관리하기에는 너무 힘든 보건관리자·간호사를 위한 경량 로컬 MSDS 관리 도구.”**

차별화 포인트는 다음과 같습니다.

| 로컬/온프레미스 | MSDS 원문 외부 업로드 없음 |
| 현장별 영향 분석 | 제품 하나가 어느 현장에 영향 주는지 자동 표시 |

## 21. 최종 제안

이 앱의 1차 목표는 다음과 같이 잡는 것이 좋습니다.

> **우리 사업장에서 보유한 MSDS를 한곳에 모으고, 제품별·현장별로 연결하며, MSDS 핵심정보를 자동 추출하고, 제출번호/개정일/CAS No./영업비밀/공급사 확인 필요 항목을 찾아 보건관리자가 검수하기 쉬운 엑셀 대장으로 만들어주는 로컬 업무보조 앱.**

한 문장으로 정리하면 다음과 같습니다.

> **이 앱은 간호사님이 MSDS를 직접 다 읽고, 엑셀에 옮기고, 개정 필요 여부를 기억하고, 공급사에 하나씩 요청해야 하는 일을 줄여주는 “MSDS 관리대장 자동화 및 변경감시 도구”입니다.**

초기 버전에서는 자동판정이나 자동등록을 목표로 하지 않고, **정리·검수·알림·엑셀화**에 집중하는 것이 가장 현실적입니다.
